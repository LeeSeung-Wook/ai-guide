# 1. 목적/범위

본 문서는 `.person/docs/prd.md`의 “심플 다중 사용자 블로그” 요구사항을 **구현 가능한 기능 명세**로 정리한다.

- 포함: 회원/인증, 게시글, 댓글, 권한/노출 규칙, 수용 기준
- 제외: UI 상세(화면 설계/디자인), 관리자 기능, 카테고리/태그, 개인 블로그 페이지, 비기능(성능/인프라/모니터링)

# 2. 용어

- **Guest(비회원)**: 로그인 없이 조회/검색만 가능한 사용자
- **Member(회원)**: 로그인 후 글/댓글 작성 및 본인 리소스 관리 가능한 사용자
- **비노출 처리**: 데이터는 존재할 수 있으나, 어떤 목록/상세/검색에서도 사용자에게 보이지 않게 처리

# 3. 역할 및 권한

| 기능 | Guest | Member |
|---|---:|---:|
| 글 목록/상세 조회 | O | O |
| 글 검색 | O | O |
| 글 작성 | X | O |
| 글 수정/삭제(본인 글) | X | O |
| 댓글 작성 | X | O |
| 댓글 삭제(본인 댓글) | X | O |
| 회원가입/로그인/탈퇴 | X(가입/로그인은 가능) | O |

> 관리자(Admin) 역할은 없다. 사용자는 **본인 리소스만** 관리한다.

# 4. 도메인(최소 모델)

PRD가 데이터 모델을 명시하지 않으므로, 기능 구현을 위해 필요한 최소 필드만 정의한다.

## 4.1 User

- `id`
- `username` (로그인 식별자, 유일)
- `password` (저장 시 해시)
- `status`: `ACTIVE` | `WITHDRAWN`
- `createdAt`, `updatedAt`

## 4.2 Post

- `id`
- `authorId`
- `title`
- `content` (텍스트)
- `status`: `ACTIVE` | `DELETED` (삭제/비노출)
- `createdAt`, `updatedAt`

## 4.3 Comment

- `id`
- `postId`
- `authorId`
- `content` (텍스트)
- `status`: `ACTIVE` | `DELETED` (삭제/비노출)
- `createdAt`, `updatedAt`

# 5. 공통 규칙

## 5.1 노출 규칙(핵심)

1. `User.status=WITHDRAWN` 인 사용자가 작성한 **모든 게시글/댓글은 즉시 비노출**된다.
2. `Post.status=DELETED` 인 게시글은 목록/상세/검색에서 비노출된다.
3. `Comment.status=DELETED` 인 댓글은 댓글 목록에서 비노출된다.
4. 비노출된 리소스에 대해 클라이언트가 직접 ID로 접근하더라도, **존재 여부가 노출되지 않도록** 기본 응답은 `404 Not Found`로 한다.

## 5.2 권한 규칙

- 로그인하지 않으면(Guest) 쓰기/수정/삭제 불가.
- 게시글/댓글의 수정/삭제는 **작성자 본인만** 가능.

## 5.3 정렬/페이징(목록 공통)

- 게시글 목록 기본 정렬: `createdAt` 내림차순(최신순)
- 기본 페이징 파라미터(권장): `page`(0부터), `size`
  - 기본값: `page=0`, `size=20`

# 6. 기능 명세

## 6.1 회원 관리 (Authentication)

### 6.1.1 회원가입

- 행위: 사용자가 `username`, `password`로 계정을 생성한다.
- 성공 결과: 신규 `User(status=ACTIVE)` 생성.
- 검증(최소):
  - `username`은 공백 불가, 중복 불가
  - `password`는 공백 불가
- 실패:
  - 중복 아이디: `400 Bad Request` (또는 `409 Conflict` 중 택1; 프로젝트 표준에 맞춰 일관 적용)

### 6.1.2 로그인

- 행위: 사용자가 가입된 정보로 인증을 수행한다.
- 성공 결과: “로그인 상태”가 된다.
  - (구현 선택) 세션 기반 또는 토큰(JWT) 기반 중 하나로 일관되게 제공
- 실패:
  - 자격 증명 불일치: `401 Unauthorized`

### 6.1.3 회원 탈퇴

- 행위: 로그인한 회원이 탈퇴를 요청한다.
- 처리:
  - `User.status=WITHDRAWN`로 변경
  - 탈퇴 즉시, 해당 사용자의 **게시글/댓글은 모든 화면에서 비노출**
- 성공 결과:
  - 게시글 목록/상세/검색에서 해당 사용자 게시글이 사라진다.
  - 게시글 상세의 댓글 목록에서 해당 사용자 댓글이 사라진다.
- 실패:
  - 비로그인 요청: `401 Unauthorized`

## 6.2 게시글 기능 (Posts)

### 6.2.1 글 목록 조회 (최신순, 페이징)

- 대상: Guest/Member
- 입력: `page`, `size`
- 출력(필수 포함):
  - 게시글 요약 목록(예: `id`, `title`, `author`, `createdAt`)
  - 페이징 정보(예: `page`, `size`, `totalElements`/`hasNext` 중 1가지 이상)
- 규칙:
  - `DELETED` 게시글 비노출
  - `WITHDRAWN` 사용자의 게시글 비노출

### 6.2.2 글 상세 조회 (본문 + 댓글)

- 대상: Guest/Member
- 입력: `postId`
- 출력:
  - 게시글 전체 본문
  - 댓글 목록(1단 구조)
- 규칙:
  - 비노출 대상(삭제/탈퇴 작성자)은 `404 Not Found`
  - 댓글 목록은 `Comment.status=ACTIVE` 이고 작성자 `ACTIVE` 인 것만 노출
  - 댓글 정렬(권장): `createdAt` 오름차순

### 6.2.3 글 작성 (텍스트 전용)

- 대상: Member
- 입력: `title`, `content`
- 규칙:
  - 이미지/에디터 기능은 제공하지 않는다(텍스트만).
  - 비로그인 요청은 `401 Unauthorized`
- 성공 결과:
  - 새 게시글 생성 및 `id` 반환

### 6.2.4 글 수정

- 대상: Member(작성자 본인)
- 입력: `postId`, 수정 필드(`title`, `content`)
- 규칙:
  - 작성자 본인만 가능(타인 시도: `403 Forbidden`)
  - 비노출 대상(삭제/탈퇴 작성자)은 `404 Not Found`

### 6.2.5 글 삭제

- 대상: Member(작성자 본인)
- 입력: `postId`
- 처리:
  - 게시글을 즉시 비노출 처리(`status=DELETED` 권장)
- 규칙:
  - 작성자 본인만 가능(타인 시도: `403 Forbidden`)
  - 비노출 대상은 `404 Not Found`

### 6.2.6 글 검색 (제목/본문 키워드)

- 대상: Guest/Member
- 입력:
  - `q` (검색어)
  - `page`, `size`
- 규칙:
  - 제목 또는 본문에 `q`가 포함된 게시글만 반환
  - 비노출 대상(삭제/탈퇴 작성자)은 결과에서 제외
- 주의:
  - 대/소문자, 공백 트리밍, 다중 키워드 등은 PRD에 없으므로 **프로젝트 표준에 맞춰 추가 정의**한다.

## 6.3 댓글 기능 (Comments)

### 6.3.1 댓글 작성

- 대상: Member
- 입력: `postId`, `content`
- 규칙:
  - 대댓글 없이 **게시글에 직접 달리는 1단 구조**
  - 비로그인 요청: `401 Unauthorized`
  - 대상 게시글이 비노출이면 `404 Not Found`

### 6.3.2 댓글 삭제

- 대상: Member(작성자 본인)
- 입력: `commentId`
- 처리:
  - 댓글을 즉시 비노출 처리(`status=DELETED` 권장)
- 규칙:
  - 작성자 본인만 가능(타인 시도: `403 Forbidden`)
  - 비노출 대상은 `404 Not Found`

# 7. 수용 기준(AC) 매핑

1. 최신순 정렬 + 페이징으로 게시글 목록을 확인할 수 있다. (6.2.1)
2. 검색어로 제목/본문 기반 필터링이 된다. (6.2.6)
3. 로그인한 사용자만 글/댓글 작성 가능, 본인 것만 관리 가능. (3, 6.2.3~6.2.5, 6.3)
4. 회원 탈퇴 시 해당 사용자의 콘텐츠가 즉시 모든 목록/상세에서 사라진다. (5.1, 6.1.3)

PRD에 명시가 없어 구현 전에 결정을 권장한다(결정 전까지는 기능 동작이 모호해질 수 있음).

- 검색 상세 규칙: 대소문자 무시 여부, 형태소/부분일치, 다중 키워드 처리
